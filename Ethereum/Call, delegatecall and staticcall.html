✅ Background: What’s the ABI?

The ABI (Application Binary Interface) in Solidity defines:

  • How data (like function names, arguments, return types) is encoded.

  • How contracts interact with each other (function calling conventions).

  • How to decode return values from functions.

When you call a contract normally (e.g., otherContract.doSomething(123);), Solidity and the EVM handle the ABI encoding/decoding automatically behind the scenes.

But some contracts:

  • Don't expose their interfaces in standard ABI-compatible ways,

  • Use unusual fallback or low-level patterns,

  • Are written in different languages (Yul, Huff, or raw EVM bytecode),

  • Or you just want to manually craft low-level calls for optimization or control.

That’s where the low-level functions come in:

💡 Use Cases “In Detail”

🧵 1. Interfacing with contracts that do NOT adhere to the ABI

Sometimes, the target contract:

  • Uses custom function selectors or fallback logic,

  • Expects inputs in non-standard formats,

  • Or hides its interface (e.g., proxies, obfuscation, or minimal contracts).

In such cases, using normal function syntax like foo.bar(x) fails — so you must use low-level functions.

🔍 Example: Interacting with a contract without an interface

solidity
(bool success, bytes memory data) = address(someContract).call(
    abi.encodeWithSignature("doSomething(uint256)", 123)
);

You are manually encoding:

  • "doSomething(uint256)" → the function selector,

  • 123 → the arguments.

Here, call is used even if the contract doesn't expose its ABI in Solidity.
This is also helpful when:

  • The interface is not available at compile time.

  • You need to interact with raw EVM code.

🧵 2. Get more direct control over encoding

Solidity's normal function calls handle encoding/decoding for you. But sometimes you want:

  • Full manual control over calldata layout (e.g., for gas optimization),

  • To dynamically construct function names or arguments,

  • Or to implement advanced patterns (like forwarding, proxies, or meta-transactions).

That’s where call, delegatecall, and staticcall shine.

🔍 Example: Dynamic function call

solidity
string memory method = "transfer(address,uint256)";
bytes memory data = abi.encodeWithSignature(method, to, amount);
(bool success, bytes memory returndata) = token.call(data);

  • You dynamically construct the calldata using abi.encodeWithSignature.

  • token.call(data) sends the raw bytes to the token contract.

  • You handle the return data manually.

🧠 What You Must Handle Yourself

When using these low-level functions, Solidity does not protect you:

  • No compile-time type checking

  • No function signature matching

  • You must manually:

    • Encode input (use abi.encodeWithSelector, etc.)

    • Decode output (abi.decode)

    • Check if the call succeeded (success)

    • Handle reverts or unexpected return data

🚨 Security Warning

Low-level calls are powerful but dangerous. They:

  • Can fail silently if success is not checked.

  • Can be used in reentrancy attacks if not carefully handled.

  • Can easily break assumptions about logic and state.

Always validate inputs/outputs and guard against malicious behavior.
